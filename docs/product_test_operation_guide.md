# Product Test Operation Guide

## 1. Scope
- This guide covers the Product Test Data Tracing System operational flow.
- `product_test_report_item` is intentionally excluded.
- Report body is generated by `product_test_release_id` lookup.

## 2. Main Tables and Purpose
- `product_test_release`: release approval and execution scope
- `product_test_target_definition`: target master definition
- `product_test_target`: target snapshot/instance by serial
- `product_test_environment_definition`: environment master definition
- `product_test_environment`: captured environment snapshot
- `product_test_case`: test case master
- `product_test_procedure`: procedure master under test case
- `product_test_run`: execution session
- `product_test_result`: case-level result record
- `product_test_procedure_result`: procedure-level result record
- `product_test_evidence`: evidence linked to result / procedure result / defect
- `product_test_defect`: defect linked to original failed result and optional retest result
- `product_test_report`: report header and approval object
- `product_test_status_transition`: lifecycle audit history

## 3. User Roles
- Admin/SQA
  - master data setup
  - release creation
  - report review / approve / reject
  - trace review / export / system check
- Tester
  - run start
  - procedure result input
  - evidence attach
  - defect registration
  - retest creation support

## 4. Standard Flow
1. Master Data setup
2. Product Test Release creation
3. Target / Environment setup
4. Run start
5. Procedure result input
6. Evidence attach
7. Defect registration
8. Retest
9. Report approval / rejection

## 5. Important Rules
- `product_test_report_item` is intentionally excluded.
- Report body is generated by `product_test_release_id` lookup.
- Failed `product_test_result` is immutable.
- Retest creates a new `product_test_result`.
- `product_test_defect` links the original failed result and the retest result.
- Status changes must be recorded in `product_test_status_transition`.
- Approved report locks source records for operational editing.

## 6. Blocked vs Skipped
- `blocked`
  - execution could not continue due to blocker
  - requires `judgement_reason`
  - examples: `environment_issue`, `target_issue`, `tool_issue`, `permission_issue`, `precondition_not_met`
- `skipped`
  - execution intentionally omitted by scope or policy
  - requires `judgement_reason`
  - examples: `out_of_scope`, `not_applicable`, `covered_by_previous_result`, `covered_by_other_test_case`, `excluded_by_sqa_decision`, `duplicate_test`

## 7. Hotfix Handling
- `HF` does not mean all tests are skipped.
- Execute impacted items, risk items, and regression items.
- Skip only true out-of-scope items with explicit `judgement_reason`.

## 8. Export and Print
- Report CSV: `/admin/product-test-reports/{product_test_report_id}/export.csv`
- Release trace CSV: `/admin/product-test-releases/{product_test_release_id}/trace/export.csv`
- Run CSV: `/admin/product-test-runs/{product_test_run_id}/export.csv`
- Print view: `/admin/product-test-reports/{product_test_report_id}/print`

## 9. Manual Smoke Checklist
1. Seed WiFi AP scenario.
2. Open Product Test Release trace.
3. Open Run detail.
4. Verify 5 procedure results.
5. Verify 2 failed procedure results.
6. Verify evidence count.
7. Verify defect count.
8. Confirm report approval blocked while defects are opened.
9. Move defects through lifecycle.
10. Create retest result.
11. Close defects.
12. Approve report.
13. Verify source data lock.
14. Export CSV.
15. Open print view.
